==================================================
ALIGNMENT REVIEW
==================================================

1. Database -> Backend

Status: mostly aligned

Aligned:
- backend reads role_permissions
- backend reads user_permissions
- backend reads agent_scope
- backend uses project_users
- backend uses support_access
- backend uses support_requests
- backend writes audit logs

Still transitional:
- backend still supports workspace_memberships.permissions_json
- backend still syncs legacy user_project_access

2. Backend -> Frontend

Status: partially aligned

Aligned:
- frontend page visibility generally follows backend role expectations
- frontend project access UI matches normalized project membership model
- support access and support request UX exist in both layers
- inbox and assignment UI generally follow backend scope rules

Not aligned:
- backend source of truth is role_permissions plus user_permissions plus agent_scope
- frontend source of truth is mostly hardcoded default role maps plus permissions_json
- frontend does not load role_permissions directly
- frontend does not load user_permissions directly

Practical effect:

If someone changes normalized permission rows in database, backend behavior changes immediately.
Frontend visibility may not reflect that change correctly.

3. Frontend -> Backend access rule mismatches found during review

Mismatch A
Issue: Workspace member listing is more open in backend than in frontend

Evidence:
- Route: backend-api\src\routes\workspaceRoutes.ts
- Service: backend-api\src\services\workspaceAccessService.ts
- Model: backend-api\src\models\workspaceMembershipModel.ts

Current behavior:
- GET /workspaces/:id/members uses requireWorkspaceAccess
- listWorkspaceMembersService() only requires workspace membership
- returned payload includes members, role, permissions_json, agent scope, and global role

Impact:
- any active workspace member can enumerate workspace users and permission data through API
- frontend hides workspace management from many roles, but API is still readable

Status:
- backend/frontend privacy mismatch

Mismatch B
Issue: Audit page gating in frontend is broader than backend authorization

Evidence:
- Frontend: frontend-dashboard\pages\audit.tsx
- Backend: backend-api\src\services\auditService.ts

Current behavior:
- frontend allows audit access when user has manage_workspace OR manage_users OR manage_permissions
- backend actually requires manage_workspace

Impact:
- some users can see the audit page in frontend but get blocked by API

Status:
- frontend/backend rule mismatch

Mismatch C
Issue: Assignment capacity candidates may mark editor/user roles as eligible

Evidence:
- backend-api\src\services\conversationAssignmentService.ts

Current behavior:
- candidate query includes roles: workspace_admin, workspace_owner, admin, editor, user, agent
- eligible_for_assignment is computed without checking ASSIGNABLE_ROLES
- final assignment path later rejects non-assignable roles through getAssignableMembership()

Impact:
- UI can receive candidates marked as eligible even though actual assignment will fail later

Status:
- backend internal consistency gap that can leak into frontend behavior

==================================================
MAIN IMPLEMENTATION GAPS
==================================================

Gap 1
Frontend permission source is not the same as backend permission source.

Frontend currently relies on:
- hardcoded default role permissions in frontend-dashboard\store\authStore.ts
- workspace membership permissions_json overrides

Backend currently relies on:
- role_permissions
- user_permissions
- membership permissions_json compatibility overrides

This is the biggest structural alignment issue.

Gap 2
No full roles and permissions administration module.

The tables exist, but the product surface for administering them does not.

Gap 3
Workspace members API is under-protected for privacy expectations.

Gap 4
Some page-level rules and API-level rules are not identical.
